### 2.2 decon_wav_fht / wav_fht：为何新跑比旧跑慢 6–7 倍？

**原因一：OS page cache 冷启动**

旧跑在 GPU3/2 上连续运行 3+ 天（~2778 h CPU 时间），FHT 数据文件（约 4.8 MB/文件 × 1300 文件 ≈ 6 GB）随着反复访问逐渐被 Linux kernel 缓存进内存（OS page cache）。重启后 page cache 清冷，每次文件读取触发实际磁盘 I/O。

**原因二：同期大内存进程竞争**

当前两个 FHT 训练进程：
- `decon_wav_fht`（PID 16374）：RSS **252 GB**
- `wav_fht`（PID 17441）：RSS **289 GB**

两进程合计占用 **541 GB 物理内存**，极大压缩 OS 可用于 page cache 的 RAM。每次训练的 `_load_compact_npz_arrays` + `_cache_put` 虽在 Python 层缓存最多 8 个文件，但 8 × 4.8 MB = 38 MB，远不够覆盖全部频繁访问的文件。

**原因三：磁盘池利用率过高**

磁盘池当前使用率 **96%**，高利用率使机械硬盘 seek time 增加，读写速度下降。

**预期**：随着新跑持续运行，page cache 逐渐回暖，step 时间将从 25 s 逐渐降回 ~4 s。

---

### 2.3 decon_npevst（71–97 s/step） vs wfs_wav（5 s/step）：为何同样形状却慢 14–19 倍？

两者 waveform 形状接近（`(17612, ~43, 2)` vs `(17612, 61, 2)`），pmt_batch 相近（200 vs 200），模型结构相同。batch size 不同（32 vs 16），steps/ep = 1073 vs 2015，但 **每步耗时**悬殊。

**根本原因：每事例 `parse_decon_npevst_hits` Python 循环**

`waveform_io.py` 中 `parse_decon_npevst_hits` 的瓶颈：

```python
for pid in np.unique(pmt):  # 最多 17612 次 Python 迭代
    idx = np.where(pmt == pid)[0]
    ...
    out[pid, :n, 0] = tt
    out[pid, :n, 1] = qq / qmax
```

每个训练样本需在 Python 层对所有 fired PMT 做循环（平均数百至数千次），再逐 PMT 填充 numpy 数组。CPU 在此阶段成为瓶颈，GPU 利用率下降。

对比 wfsampling：预处理时已提取 key points 并按 PMT 存储偏移（`offsets`），训练时仅需一次数组切片，无 Python 循环。

**decon_npevst 文件格式代价：**
- 存储为 pickle jagged array（`n_events × variable_hits × 3 cols`）
- 反序列化本身慢于连续 float32 数组
- 无法利用 numpy 向量化，CPU 单核成为限速步骤

### 2.4 减少 FHT 训练时间方案

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| 等待 page cache 回暖 | 无代码改动，自然优化 | 无法保证（内存压力大）；前几 epoch 极慢 | ★★☆ |
| 增大 `--cache-files`（如 100） | 代码一行改参数，覆盖更多文件 | RSS 进一步膨胀（已 252–289 GB），可能触发 OOM 或加剧 page eviction | ★☆☆ |
| 增大 `--pmt-batch-size`（150→200） | 减少 CNN 子批次（118→88），与 wfs_wav 对齐 | 可能 GPU OOM（FHT 300 样本 × 200 PMT vs 61 × 200）；需验证 | ★★☆ |
| 从 FHT 数据预提取 WFS 风格 key points | 彻底消除 disk I/O 瓶颈，训练降至 ~3 h/ep | 需新增预处理步骤；改变特征表示（不再是完整 300 样本窗口） | ★★★ |
| 合并 FHT npz 文件（减少文件数） | 减少磁盘 seek；提升读取连续性 | 需重新预处理；实现复杂 | ★★☆ |

**推荐**：短期可尝试 `--pmt-batch-size 200` 验证 GPU OOM；长期若效果可接受，考虑对 FHT 数据做 WFS-style 预处理，训练速度预计改善 5–6 倍。

### 2.5 能否用 wfs_wav 方法替代 decon_npevst 训练？

**可以，且已有数据**：`fea6+wfs_decon_waveform` 使用的 `WFSampling_decon_waveform/` 数据已于 2026-06-19 前全部完成预处理（numu 1302 / nue 1712 / nc 1428 文件）。该实验：

- 输入格式与 `fea6+wfs_wav` 相同（WFS key points，文件已按 PMT offset 索引）
- 训练速度预计 **~3 h/ep**（与 wfs_wav 一致）
- 物理含义：WFS 从 decon_waveform 提取 key points，保留主要脉冲结构，但舍弃了 decon_npevst 中的原始 npe/time 精确信息

**结论**：若接受 WFS 特征表示，建议优先启动 `fea6+wfs_decon_waveform` 代替 `fea6+decon_npevst`，可将每 epoch 训练时间从 ~21–29 h 降至 ~3 h。
